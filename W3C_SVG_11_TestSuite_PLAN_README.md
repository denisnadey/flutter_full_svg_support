# W3C SVG 1.1 Test Suite Integration Plan for browser-vs-library Screenshot Testing

## Goal

Integrate `W3C_SVG_11_TestSuite` into the current comparison pipeline:

- Browser render (Chrome/Puppeteer)
- Flutter render (`AnimatedSvgPicture`)
- Pixel diff (`tool/golden_capture/image_compare.dart`)

And obtain a scalable, deterministic, CI-friendly test suite.

## Implementation Status (2026-04-10)

- Stage 1 implemented: manifest is generated by the `tool/w3c_goldens/generate_manifest.js` script.
- Stage 2 implemented: browser capture for W3C works via a local HTTP server (`capture_browser_w3c.js`).
- Stage 3 implemented: `test/golden_comparison/w3c_golden_comparison_test.dart` reads the manifest, renders, compares, and saves artifacts.
- Orchestrator `tool/w3c_goldens/run_w3c_comparison.sh` added with flags:
  - `--enable-render`
  - `--no-enforce-threshold`
  - `--debug-trace`

Current focus:

- stabilizing the smoke pool and calibrating thresholds;
- after stabilization, enable strict mode (`W3C_ENFORCE_THRESHOLD=true`) for CI smoke.

## Plan to Achieve All W3C Tests Passing

Below is the plan specifically for the goal "all tests pass".

Working assumption:

- for the screenshot pipeline "passing" = deterministic render and similarity to the browser baseline above the threshold;
- interactive/DOM/script cases are separate, because they are not equivalent to a regular static screenshot.

### Stage 0. Lock Down the Target Scope and Pass Criterion

Tasks:

1. Finalize the list of cases:
   - `approved` visual,
   - `approved` animation,
   - `approved` script/interact (separate track).
2. Define the pass metric:
   - `similarity >= threshold`,
   - `no timeout`,
   - `no parser/runtime crash`.
3. Introduce a single report artifact (`json + markdown`) for each run.

Readiness criterion:

- there is one source of truth for pass/fail and coverage across cases.

### Stage 1. Full Diagnostic of 100% of the Suite

Tasks:

1. Run all cases in the mode:
   - `--enable-render`,
   - `--no-enforce-threshold`.
2. Build a problem map by category:
   - crash/timeout,
   - empty render,
   - transform mismatch,
   - gradient/pattern mismatch,
   - clip/mask mismatch,
   - filters mismatch,
   - text/fonts mismatch,
   - animation mismatch,
   - script/interact mismatch.
3. For each category identify the top-10 reproducer cases.

Readiness criterion:

- there is a prioritized backlog based on real root causes, not individual cases.

### Stage 2. Close Stability Blockers (mandatory priority)

Tasks:

1. Completely eliminate timeout/hang/crash for all approved cases.
2. Add a fail-fast guard on heavy steps (capture/compare/decode) with an explicit error.
3. Add regression tests for every type of hang found.

Readiness criterion:

- 0 hangs in the full run, only valid pass/fail results.

### Stage 3. Resources, References, and Fonts (largest source of discrepancies)

Tasks:

1. Implement a correct resolver for `../resources/*` and `../images/*`.
2. Align handling of external references (`xlink:href`, `href`, `url(#id)`).
3. Introduce a font strategy:
   - load W3C test fonts where possible,
   - fallback policy to preserve determinism.
4. Recalculate thresholds after text stabilization.

Readiness criterion:

- text/resource cases stop producing systematic false-fail results en masse.

### Stage 4. Fix Waves by Specification Subsystems

Wave A (geometry):

- `coords`, `paths`, `shapes`, `transform`.

Wave B (paint/model):

- `painting`, `pservers`, `gradients`, `patterns`, `opacity`.

Wave C (clip/mask/filter):

- `clipPath`, `mask`, `filter chain`, color spaces.

Wave D (text):

- text layout, anchors, baseline, spacing, textPath.

Wave E (animation):

- fixed time-snapshots for each animated case.

For each wave:

1. Take top failing cases.
2. Create minimal unit/regression tests in the library.
3. Fix the renderer.
4. Re-run the full subset for that wave.

Readiness criterion:

- a wave is closed only when there are no new regressions in already-green groups.

### Stage 5. Script/Interact Track (separate from static screenshots)

Tasks:

1. Separate script/interact cases into a dedicated manifest tier.
2. Build an event-driven harness for them (not just a single screenshot).
3. Introduce separate pass rules:
   - state after events,
   - DOM/event side effects (where applicable).

Readiness criterion:

- script/interact cases do not block the regular visual pipeline and have their own transparent status.

### Stage 6. Strict CI Gates up to 100%

PR CI:

1. smoke strict (`W3C_ENFORCE_THRESHOLD=true`).
2. no-new-failure policy on stabilized cases.

Nightly CI:

1. full approved strict.
2. auto-publish report and top regressions.

Release gate:

1. 100% pass on the target scope.
2. 0 timeout/hang.
3. 0 unexplained flaky cases over N consecutive nightly runs.

## Practical Roadmap (8 weeks)

Week 1:

1. Full-diagnostic run.
2. Triage board + grouping.
3. Finalize pass metrics.

Weeks 2-3:

1. Blockers stability.
2. Resource/font resolver.
3. Re-run full.

Weeks 4-5:

1. Wave A + B.
2. Close major visual mismatches.

Week 6:

1. Wave C.
2. Filters/masks/clipPath parity.

Week 7:

1. Wave D (text).
2. Recalibrate thresholds.

Week 8:

1. Wave E + script/interact baseline.
2. Enable full strict gate in CI.

## Commands for Executing the Plan

```bash
./tool/w3c_goldens/run_w3c_comparison.sh \
  --tier all \
  --enable-render \
  --no-enforce-threshold

# Strict smoke gate for PRs
./tool/w3c_goldens/run_w3c_comparison.sh \
  --tier smoke \
  --enable-render
```
